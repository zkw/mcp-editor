# 锚点补全代码编辑工具 产品需求与技术实现规范

## 一、项目背景与核心定位
### 1.1 行业痛点
当前LLM对已有文件的编辑，主流采用**全量重写、Unified Diff标准补丁、Search-and-Replace搜索替换块**三种方案，在实际工程落地中存在无法规避的核心瓶颈：
- 代码块重排、大规模重构场景下，Diff/补丁格式需要LLM精准追踪行号与上下文位置，极易出现错位、编辑失效；
- 基于行号的Diff方案，单次多修改场景会出现严重的行号漂移，前序修改直接导致后序补丁失效；
- 全量重写方案正确率最高，但Token成本极高、生成延迟长，大文件场景极易触发LLM“偷懒”（自行省略未修改代码）；
- LLM逐字输出未修改代码时，易因采样随机性引入拼写错误、逻辑幻觉，造成原有代码非预期篡改。

### 1.2 核心解决方案
本工具打造**基于锚点补全的伪全量重写**编辑范式，核心逻辑为：
LLM无需逐字输出未修改的文件内容，仅需使用严格连续的6个半角句号`......`作为占位符，通过`<前缀锚点代码>......<后缀锚点代码>`的语法结构，即可让工具自动从源文件中精准匹配并补全锚点之间的原有内容。
核心约束：`前缀锚点+......+后缀锚点`的组合模式，必须在源文件中全局唯一。

## 二、方案核心优越性
1.  **彻底攻克代码重排难题**
    将传统Diff方案“面向差异的修改”，转化为“面向结构拓扑的声明式编排”。代码重排操作仅需LLM调整锚点+占位符的输出顺序，无需处理复杂的删除-插入指令，大幅降低LLM的认知负担，彻底解决重排场景编辑失效的行业痛点。

2.  **极致的Token经济学，根治LLM偷懒问题**
    保留全量重写的高正确率优势，同时通过`......`占位符省略所有未修改的长文本，实现Output Token量断崖式下降，成倍降低API成本与生成延迟；同时未修改内容无需LLM生成，从根源上规避了大文件场景下LLM自行省略代码的偷懒行为。

3.  **完全免疫行号漂移，鲁棒性极强**
    采用基于内容的锚点匹配，完全解耦对文件物理行号的依赖。只要锚点组合在源文件中唯一，无论文件此前发生过多少次增删改，工具都能精准定位目标区间，单次对话多修改场景无任何行号失效风险。

4.  **物理隔离未修改内容，零幻觉、零篡改风险**
    占位符对应的原有内容，由工具在本地直接从源文件无损拷贝，无需LLM逐字输出，实现未变更代码的物理隔离，100%杜绝LLM拼写错误、自作主张优化原有逻辑带来的幻觉与非预期篡改。

## 三、核心语法与约束规则
### 3.1 核心语法定义
- 占位符：严格使用**6个连续半角句号`......`**，禁止多写、少写、使用全角句号或其他省略号字符。
- 匹配规则：引擎对`前缀锚点......后缀锚点`执行**非贪婪最短匹配**，保留前缀、后缀锚点原文，将两者之间的源文件内容自动填入占位符位置。
- 编辑逻辑：LLM输出完整的新文件结构，仅用`......`替代未修改的代码段，工具最终输出补全后的完整文件。

### 3.2 核心约束红线
1.  锚点必须与源文件对应内容**100%字节级一致**，严禁修改缩进、增删空格、换行符或字符内容；
2.  `前缀锚点+......+后缀锚点`的组合，必须在整个源文件中**全局唯一匹配**，禁止使用通用字符（如单独的`{`、`}`、`return`）作为锚点；
3.  锚点需具备高特征熵，优先使用完整函数声明、特有注释、连续多行上下文作为锚点，确保唯一性；
4.  锚点配对必须满足`前缀锚点结束位置 < 后缀锚点起始位置`，禁止锚点重叠。

## 四、技术实现详细规范
### 4.1 核心数据结构设计
| 数据结构 | 设计规范 | 核心作用 |
| :--- | :--- | :--- |
| 一维连续字符缓冲区 | 将源文件完整加载为单一字符串，保留所有`\n`换行符与空白字符 | 所有子串查找、正则匹配均在此一维空间执行，保证最高的搜索效率 |
| 行偏移量数组 | int[]/List<int>类型，记录源文件每一行行首字符在一维缓冲区中的绝对索引，示例：`[0, 15, 32, ...]` | 通过二分查找，O(logN)时间内将任意字符索引转换为对应行号，用于错误反馈时快速定位锚点位置 |
| Piece Table片段表 | 可选，高频/大文件编辑场景启用，维护字符串指针而非直接操作内存字符串 | 实现O(1)/O(logN)级别的任意位置插入、删除，大幅提升大文件修改的执行效率 |

### 4.2 核心匹配算法（最短有效区间搜索算法）
#### Step1：锚点独立搜索
1.  从LLM输出内容中，提取所有`......`对应的前缀锚点A、后缀锚点C；
2.  使用工业级子串查找算法（Boyer-Moore，或编程语言底层高度优化的`str.find`/`indexOf`），在一维缓冲区中分别查找A、C的所有出现位置；
3.  输出结果：前缀锚点起始位置数组`starts_A = [a1, a2, a3...]`，后缀锚点起始位置数组`starts_C = [c1, c2, c3...]`。

#### Step2：合法区间懒惰配对
1.  遍历`starts_A`中的每一个起始点`ai`；
2.  对每个`ai`，通过二分查找在`starts_C`中，找到**大于等于`ai + length(A)`的第一个元素`cj`**；
3.  若找到符合条件的`cj`，则`(ai, cj)`为一个潜在合法匹配区间，记录所有合法区间组合。

#### Step3：唯一性判定
1.  合法区间数量`Count == 1`：锚点组合全局唯一，匹配成功，进入文件替换流程；
2.  合法区间数量`Count == 0`：无有效配对，触发歧义回退；
3.  合法区间数量`Count > 1`：锚点组合存在歧义，触发歧义回退。

### 4.3 文件替换执行流程
1.  匹配成功后，确定有效区间`(ai, cj)`，其中锚点A的结束位置为`ai + length(A)`，锚点C的起始位置为`cj`；
2.  保留LLM输出的前缀锚点A、后缀锚点C原文；
3.  从源文件一维缓冲区中，提取`[ai + length(A), cj]`区间的完整内容，填入`......`占位符位置；
4.  按LLM输出的文件拓扑顺序，完成所有占位符的补全，最终输出完整的新文件。

### 4.4 歧义回退与错误反馈机制
#### 4.4.1 错误信息生成流程
1.  遍历`starts_A`和`starts_C`，通过行偏移量数组的二分查找，将每个锚点的字符索引转换为对应行号；
2.  根据行号提取锚点所在的完整行文本（锚点跨越多行时，提取首行与末行）；
3.  按照固定格式组装结构化错误信息，返回给LLM。

#### 4.4.2 标准错误返回格式
```
[System Error] Pattern ambiguity detected.
The prefix "【前缀锚点内容】" and suffix "【后缀锚点内容】" combination is NOT unique or cannot be resolved.

Found "【前缀锚点内容】" at the following lines:
- Line 【行号】: `【锚点所在完整行文本】`
- Line 【行号】: `【锚点所在完整行文本】`

Found "【后缀锚点内容】" at the following lines:
- Line 【行号】: `【锚点所在完整行文本】`
- Line 【行号】: `【锚点所在完整行文本】`

Action Required: Please provide longer, more specific context for the anchors (e.g., include the lines above the def or below the return) so the `【前缀锚点】......【后缀锚点】` pattern matches exactly ONE block in the file.
```

## 六、边界场景与容错处理规范
1.  **通用锚点截断防御**
    若锚点匹配结果超过预设上限（建议最多返回前5个匹配项），错误反馈中仅返回前5条，避免庞大的错误内容撑爆LLM上下文窗口。

2.  **锚点重叠处理**
    配对逻辑中严格执行`ai + length(A) ≤ cj`的校验规则，天然过滤锚点重叠的非法配对，直接触发无匹配结果回退。

3.  **空白字符兼容处理**
    可选支持宽容匹配模式：解析锚点时，忽略首尾空白字符，或对缩进、换行符做正则兼容处理，适配LLM偶发的空格输出偏差。

4.  **弹性唯一性校验（可选增强功能）**
    锚点组合不唯一时，可通过锚点在新文件中的相对位置、最长公共子序列（LCS）算法，在局部范围推断最优匹配项，降低LLM修正成本。

## 七、开发优先级建议

2.  **P1基础体验**：标准错误反馈机制、多锚点对批量处理、边界场景基础防御；

4.  **P3生态适配**：LLM提示词调优、主流AI Agent框架适配、IDE插件集成能力。

# 附录A 分层递归折叠读取器设计规范
本附录为锚点补全代码编辑工具的配套读取工具设计规范，与正文的锚点补全编辑协议形成**完整的LLM文件读写闭环**，专门解决超长代码文件的读取、全局结构理解与上下文窗口适配痛点，与锚点补全编辑器共用同一套语义规范，实现读写操作的无缝协同。

## A.1 工具核心定位与设计逻辑
### A.1.1 核心定位
针对超过8K Token阈值的超长代码文件，打造**分层递归折叠读取器（Recursive Structural Folding Reader）**。在不丢失文件全局拓扑结构的前提下，通过结构化贪心折叠算法，将单次给LLM的读取内容控制在8K Token以内；同时支持LLM按需逐层展开折叠内容，兼顾全局架构认知与局部细节获取。

### A.1.2 核心算法与规则
1.  **统一语义规范**：折叠占位符与锚点补全编辑工具100%对齐，严格使用**6个连续半角句号`......`**作为折叠标识，语义完全统一。
2.  **可折叠范围**：文件中所有`()`、`[]`、`{}`配对括号之间的内容，均为可折叠对象。
3.  **贪心折叠核心执行逻辑**：
    - 优先级规则：优先折叠当前代码中**最长的、内部无嵌套同类型括号**的配对括号块；
    - 无括号长文本压缩：对于没有任何配对括号的超长文本，也会将该段视为长度 n 的整体，按约 `sqrt(n)` 进行等长切分，并将这些小段与括号块一起进入折叠竞争；这样即便没有括号，长文本也能被分段压缩。
    - 迭代执行：重复上述折叠操作，直到整个文件的Token大小不超过8K阈值，停止折叠；
    - 格式约束：仅折叠括号内的内容，保留括号首尾的锚点特征，折叠后固定格式为`前缀锚点......后缀锚点`，与编辑协议完全对齐；相邻折叠块最终会合并成单个`......`占位符，避免产生连续多个占位符。
4.  **按需展开规则**：
    - LLM可通过`Aaaaa......Ccccc`格式的参数，请求展开对应折叠块的完整内容；
    - 展开后的内容若仍超过8K Token阈值，将按照相同的贪心折叠规则进行二次折叠；
    - 支持LLM通过多层级展开请求，以“剥洋葱”的方式逐层获取代码细节，直到拿到完整的目标代码块。

## A.2 方案核心优越性论证
### 1. 结构化注意力引导，根治长文本“中间迷失”问题
LLM处理超长文本时存在显著的“Lost in the Middle”现象，对文本中间段落的注意力与理解精度会大幅下降。本方案的贪心折叠算法，本质是自动化的代码结构化摘要：通过折叠最深层、最长的非嵌套括号块，隐藏代码的“叶子节点”（具体实现逻辑、长配置项等），优先暴露代码的“骨架节点”（类定义、函数签名、控制流框架），强制LLM先从宏观架构层面建立完整的代码认知，再按需深入局部细节，从根源上避免模型陷入细节泥潭而丢失全局视野。

### 2. 极致的Token效率，实现代码内容按需懒加载
传统的滑动窗口读取方案会丢失全局拓扑结构，而全量读取则会造成巨额Token浪费与推理延迟。本方案通过8K Token的阈值控制，完美适配主流LLM的标准上下文窗口，保证稳定的推理响应速度；同时通过分层折叠的设计，实现了代码内容的**按需懒加载（Lazy Loading）**——LLM仅在需要理解对应逻辑时，才消耗Token展开对应内容，将无效Token消耗降到最低，完美平衡了全局结构视野与推理成本控制。

### 3. 读写语义完全对称，大幅降低定位幻觉风险
这是本套工具体系最核心的设计优势：读取工具与正文的锚点补全编辑工具，采用**完全一致的语义协议与占位符规范**，形成了完美的读写闭环。
LLM读取时看到的是`Aaaaa......Ccccc`格式的代码块边界，修改时提交的是同格式的编辑指令，读取阶段建立的代码块边界认知，可直接复用于编辑阶段。这种读写坐标系的完全统一，极大降低了LLM在代码定位时的幻觉风险，从底层提升了编辑操作的准确率。

### 4. 贪心折叠算法的天然合理性，兼顾骨架完整性与细节可控性
以“最长的、内部无嵌套同类型括号的块”为折叠优先级的贪心规则，具备极强的工程合理性：优先折叠的内容，通常为具体算法实现、长配置项、大规模数据初始化等叶子节点逻辑；而`for/while`循环、`if/else`判断等核心控制流骨架会被优先保留，保证折叠后的代码始终具备完整的逻辑拓扑。同时分层折叠的设计，让LLM获取的信息始终维持在8K左右的最优推理区间，使其始终处于高信息密度、低噪音的理想推理状态。

## A.3 与编辑工具的协同规范
1.  协议完全对齐：本工具的占位符格式、锚点定义、语法规范与锚点补全编辑工具100%统一，无任何格式差异；
2.  锚点信息复用：LLM在读取阶段获取的锚点特征，可直接用于编辑阶段的锚点补全指令，无需额外格式转换；
3.  闭环流程适配：LLM可通过本工具完成代码全局结构读取→按需展开目标代码块→使用锚点补全协议提交编辑指令的全流程操作，实现无断点的LLM代码编辑全链路。
